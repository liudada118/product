调研目标：明确“有了数据对比之后，如何生成分析报告”，让数据对比结果可以被研发、科研、销售演示和客户交付使用。

## 1. 结论先行

数据对比之后的报告不应该只是“导出一张截图”，而应该是把以下内容组织成一个可交付文档：

| 报告组成 | 说明 |
|---|---|
| 测试背景 | 谁测的、什么设备、什么对象、什么时间、什么模式 |
| 对比对象 | A/B 两组数据，或同一记录内 A/B 时间点、时间段 |
| 关键图像 | A 压力图、B 压力图、B-A 差值图 |
| 核心指标 | 平均压强、最大压强、最小压强、压力总和、受压面积、有效点数、重心 |
| 变化结论 | B 相比 A 增加/降低了多少，变化率多少 |
| 框选区域 | 如果有框选，输出每个区域的局部对比结果 |
| 曲线趋势 | 压力总和曲线、面积曲线、重心轨迹，P1/P2 支持 |
| 数据口径 | 采样率、置零、翻转、噪点消除、单位、方向配置 |
| 附录数据 | CSV 原始数据或指标明细 |

推荐路线：

| 阶段 | 能力 | 建议 |
|---|---|---|
| V2.0 | 对比结果摘要 + 当前帧报告预览 | 先做轻量闭环 |
| V2.1 | PDF/Word 标准分析报告 | 用于客户交付和研发留档 |
| V2.2 | 报告模板、批量报告、品牌化报告 | 用于规模化交付 |

## 3. 报告能力定位

| 项目 | 定义 |
|---|---|
| 功能名称 | 数据对比分析报告 |
| 功能定位 | 把数据对比结果转成可阅读、可交付、可追溯的分析文档 |
| 使用入口 | 数据对比页、历史回放页、框选分析结果页 |
| 目标用户 | 汽车座椅研发工程师、科研客户、展厅/门店演示人员、内部测试工程师 |
| 核心价值 | 不只是看图，而是形成“对比对象、差异、原因线索、数据证据”的闭环 |
| 不建议做的事 | V2.0 不做自动医学/健康诊断，不输出“好/坏”绝对判断，只输出数据变化和风险提示 |

| 报告类型 | 使用场景 | 内容深度 | 推荐版本 |

| 科研详细报告 | 论文、实验、招标、研发验证 | 完整指标、时间曲线、区域统计、导出附录 | V2.1/V2.2 |
| 批量测试报告 | 多组座椅、多次测试、批量样品 | 多条记录汇总、排名、异常项 | V2.2 |
| 品牌化交付报告 | 给客户或渠道 | 公司 logo、封面、结论页、可隐藏技术细节 | V2.2 |

## 5. 报告内容结构

### 5.1 标准报告目录

| 序号 | 章节 | 内容 | 是否建议 V2.0 |
|---|---|---|---|
| 1 | 报告封面 | 报告名称、测试对象、测试时间、软件版本、导出时间 | P1 |
| 2 | 测试信息 | 设备类型、MAC、传感器类型、对象、采样率、测试人员 | P0 |
| 3 | 对比说明 | 对比模式、A/B 数据来源、A/B 时间点或时间段 | P0 |
| 4 | 核心结论摘要 | 最大变化项、压力集中变化、面积变化、重心迁移 | P0 |
| 5 | A/B/差值图 | A 压力图、B 压力图、B-A 差值图 | P0 |
| 6 | 指标对比表 | A 值、B 值、差值、变化率 | P0 |
| 7 | 框选区域分析 | 每个框选区域的局部指标和差值 | P0/P1 |
| 8 | 趋势曲线 | 压力总和、面积、重心轨迹 | P1 |
| 9 | 数据口径说明 | 置零、翻转、噪点消除、单位、方向 | P0 |
| 10 | 附录 | CSV 字段说明、原始数据文件路径、备注 | P1 |

### 5.2 核心结论摘要

| 结论项 | 示例文案 | 数据来源 |
|---|---|---|
| 压力总和变化 | B 相比 A 压力总和增加 12.4% | 指标差值 |
| 最大压强变化 | 最大压强由 86 提升至 103，增加 17 | 最大值 |
| 最大值位置变化 | 最大压力点由 row=18,col=22 移动到 row=20,col=25 | 最大值坐标 |
| 受压面积变化 | 受压面积减少 8.6%，说明压力更集中 | 面积 |
| 重心变化 | 压力重心向右后方移动 | 重心坐标 |
| 框选区域变化 | 坐垫左前区域压力总和下降 15.2% | 框选指标 |

说明：V2.0 可以先用规则生成“数据变化摘要”，不要直接输出“舒适/不舒适”“合格/不合格”等结论，除非后续有明确阈值标准。

## 6. 数据对比到报告的流程

| 步骤 | 用户操作 | 系统逻辑 | 输出 |
|---|---|---|---|
| 1 | 完成数据对比 | A/B 已通过可比性校验 | 得到 A/B/差值图和指标 |
| 2 | 点击“生成报告” | 打开报告预览 | 进入报告配置页或统一分析侧栏“报告”标签 |
| 3 | 选择报告类型 | 快速报告/标准报告/详细报告 | 决定报告章节 |
| 4 | 填写报告信息 | 报告名称、测试对象、测试人员、备注 | 写入报告 metadata |
| 5 | 选择包含内容 | 图像、指标、框选、曲线、数据口径、附录 | 生成报告内容 |
| 6 | 预览报告 | 渲染报告页面 | 用户检查图和表 |
| 7 | 导出报告 | 选择 PDF/Word/HTML/图片/CSV 附件 | 生成文件 |
| 8 | 打开文件/文件夹 | 调用系统打开路径 | 完成交付 |

## 7. PM 视角：5 个产品方案

| 编号 | 方案 | 价值 | 风险 |
|---|---|---|---|
| PM-01 | 一键生成对比摘要 | 用户无需手动解读所有指标，快速知道差异 | 摘要规则必须保守 |
| PM-02 | 报告模板 | 展厅、研发、科研使用不同模板 | 模板管理会增加复杂度 |
| PM-03 | 框选区域报告 | 强化局部压力分析，突出 JQ Tools 差异化 | 多框选内容容易过多 |
| PM-04 | 报告附带 CSV | 满足科研客户复核和二次分析 | 文件管理和路径引用要清楚 |
| PM-05 | 报告历史管理 | 生成过的报告可重新打开、重命名、删除 | 需要报告存储结构 |

## 8. 设计师视角：5 个界面方案

| 编号 | 方案 | 交互说明 | 优点 | 风险 |
|---|---|---|---|---|
| UX-01 | 报告预览页 | 点击生成报告后进入独立预览页 | 可编辑、可检查、可导出 | 页面开发量中等 |
| UX-02 | 统一分析侧栏“报告”标签 | 在数据对比页右侧侧栏配置报告 | 不破坏当前对比上下文 | 侧栏空间有限 |
| UX-03 | 报告章节勾选 | 用户勾选封面、图像、指标、框选、曲线、附录 | 灵活，适合不同客户 | 选项过多需默认模板 |

## 9. 工程师视角：5 个实现方案

| 编号 | 方案 | 技术说明 | 成本 |
|---|---|---|---|
| ENG-01 | HTML 报告预览 + 浏览器打印 PDF | 前端生成 HTML 报告页，Electron 调用打印为 PDF | 低-中 |
| ENG-02 | docx 模板生成 Word | 使用模板占位符填充图像和表格 | 中 |
| ENG-03 | 报告 JSON 中间层 | 先生成 reportData，再渲染为 HTML/PDF/Word | 中，但最稳 |
| ENG-04 | Canvas/WebGL 截图 | 将 2D/3D 当前画面、A/B/差值图导出为 PNG | 中 |
| ENG-05 | CSV 附件包 | 报告旁边同时导出指标 CSV 或原始数据 CSV | 低 |

推荐技术路线：先做 `reportData JSON -> HTML 预览 -> PDF 导出`，Word 放到后续版本。这样对 Electron 应用更可控，也能先满足客户交付。

## 10. 推荐 Top 5 功能

| 排名 | 功能 | 一句话描述 | 为什么选 | 关键假设 |
|---|---|---|---|---|
| 1 | 对比报告 JSON 中间层 | 把 A/B 数据、图像、指标、框选、备注统一成 reportData | 后续 PDF、Word、HTML 都能复用 | 研发愿意先做数据结构 |
| 2 | 标准对比报告预览 | 在导出前展示报告封面、图像、指标和摘要 | 用户能检查报告是否正确 | 预览性能可接受 |
| 3 | A/B/差值图自动入报 | 自动截取当前 A 图、B 图、差值图 | 报告最核心的可视证据 | 图像导出清晰度足够 |
| 4 | 指标变化摘要 | 自动生成差值和变化率摘要 | 降低用户解读成本 | 文案只描述数据，不做过度判断 |
| 5 | 框选区域报告 | 每个框选区域输出局部对比表和图 | 突出局部分析价值 | 多框选报告布局清楚 |

## 15. 需要产品决策的问题

| 编号 | 决策项 | 推荐方案 | 备选方案 | 影响 |
|---|---|---|---|---|
| DEC-RPT-01 | V2.0 是否做完整报告导出 | 不做完整 PDF/Word，只做报告数据和预览能力 | 直接做 PDF | 直接做 PDF 会拉长周期 |
| DEC-RPT-02 | V2.1 首选导出格式 | PDF | Word、HTML、长图 | PDF 最适合交付 |
| DEC-RPT-03 | 报告是否允许用户编辑摘要 | 允许编辑并保存 | 不允许编辑 | 允许编辑更适合客户交付 |
| DEC-RPT-04 | 报告是否包含 3D 图 | V2.1 可选包含当前 3D 截图 | 只包含 2D/差值图 | 3D 截图实现和版式复杂一些 |
| DEC-RPT-05 | 报告是否包含完整 CSV 附件 | 默认不内嵌，提供同目录 CSV 附件 | 内嵌全部数据 | 内嵌会让报告过大 |
| DEC-RPT-06 | 是否生成自动结论 | 只生成数据变化摘要，不生成判断性结论 | 生成“优/差/风险”判断 | 判断性结论需要阈值和行业标准 |
| DEC-RPT-07 | 报告是否保存历史 | 保存报告记录和导出路径 | 只导出文件不入库 | 保存历史便于追溯 |

## 16. 推荐的最小可行方案

V2.0 不建议直接做复杂报告系统。建议先补“报告数据层”和“报告预览入口”，为 V2.1 PDF 做准备。

| 能力 | 说明 |
|---|---|
| 报告入口 | 数据对比页增加“生成报告”按钮 |
| 报告预览 | 显示报告标题、测试信息、A/B/差值图、指标表、框选指标、数据口径 |
| 报告数据 | 生成 reportData JSON，可保存到历史或临时缓存 |
| 导出能力 | V2.0 可先导出 HTML 或截图；PDF 放 V2.1 |
| CSV 附件 | 复用现有 CSV 导出能力，和报告同目录保存 |

| 14.8 报告异常处理 | 图片生成失败、路径错误、字段缺失 |
| 14.9 报告验收标准 | 可预览、可导出、字段可追溯 |

## 18. 总结

数据对比解决的是“差异在哪里”，分析报告解决的是“如何把差异交付给别人”。对于 JQ Tools，报告能力不应该一开始做得很重。

推荐顺序：

| 阶段 | 重点 |
|---|---|
| 第一步 | 把对比结果结构化成 reportData |
| 第二步 | 做报告预览页，保证图、表、摘要和数据口径准确 |
| 第三步 | 导出 PDF，并同目录导出 CSV 附件 |
| 第四步 | 做模板、Word、批量和品牌化 |

这样既能补齐产品闭环，又不会在 V2.0 阶段把研发复杂度拉得过高。

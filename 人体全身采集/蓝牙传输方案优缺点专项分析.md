# 蓝牙传输方案优缺点专项分析

> 项目：人体全身压力采集  
> 版本：v0.1  
> 日期：2026-05-17  
> 结论：蓝牙适合做设备配置、状态监控、低频预览和移动端连接；不建议作为 100Hz、1024 点位全量压力原始数据的主采集链路。第一版主链路仍建议用 Ethernet，蓝牙可以作为辅助链路。

## 1. 一句话结论

如果你们要稳定采集：

```text
100Hz x 1024 点位 x 连续动作数据
```

蓝牙不是最合适的主传输方案。

建议产品链路：

```text
主数据链路：Ethernet
辅助控制链路：Bluetooth LE
```

也就是说：

- 压力原始数据走 Ethernet。
- 手机配网、设备状态、电量、开始/停止、低频预览可以走 Bluetooth LE。

## 2. 数据量估算

你们的数据规模：

| 参数 | 数值 |
| --- | ---: |
| 采样率 | 100Hz |
| 点位数 | 1024 |
| 压力精度 | 0.01N |
| 每秒帧数 | 100 帧 |

如果每个点位用 `uint16` 表示：

```text
1024 点 x 2 bytes = 2048 bytes/帧
2048 bytes x 100Hz = 204,800 bytes/s
约 200 KB/s
约 1.64 Mbps
```

如果每个点位用 `float32` 表示：

```text
1024 点 x 4 bytes = 4096 bytes/帧
4096 bytes x 100Hz = 409,600 bytes/s
约 400 KB/s
约 3.28 Mbps
```

再加上：

- 时间戳
- frame_id
- 设备状态
- CRC
- 协议开销
- 重传/丢包处理

真实链路最好要有明显余量。

对 Ethernet 来说，这个数据量很小。  
对 Bluetooth LE 来说，这已经是比较吃力的连续流。

## 3. 蓝牙类型区分

### 3.1 Bluetooth Classic

常用于：

- 蓝牙耳机
- 音箱
- 串口 SPP
- 传统音频

优点：

- 传统串口模型好理解。
- 某些模块开发简单。

缺点：

- iOS 对 Classic SPP 支持受限。
- 不适合现代低功耗传感器生态。
- 功耗通常比 BLE 高。
- 医疗/穿戴设备更常用 BLE。

### 3.2 Bluetooth LE / BLE

常用于：

- 手环
- 传感器
- 医疗健康设备
- 设备配置
- 手机 App 连接

优点：

- 低功耗。
- 手机兼容性好。
- 适合控制和小数据量。
- 生态成熟。

缺点：

- 连续高吞吐数据不稳定。
- 吞吐受手机、系统、连接参数、MTU、PHY、环境影响很大。
- 调试复杂。
- 多设备并发更麻烦。

### 3.3 BLE 2M PHY

Bluetooth Core 5.0 引入 LE 2M PHY。Bluetooth SIG 资料说明，LE 2M PHY 物理层符号率为 2 Ms/s，并能提高数据速率。

但注意：

> 2M PHY 不等于应用层稳定可用 2Mbps。

实际应用吞吐会受：

- 连接间隔
- ATT MTU
- Data Length Extension
- 手机系统限制
- BLE 芯片能力
- 2.4GHz 干扰
- 距离和人体遮挡

所以不能按理论值做主链路设计。

参考：

- Bluetooth Core 6.0 specification：https://www.bluetooth.com/specifications/specs/core-specification-6-0/
- Bluetooth Core 5.0 LE 2M PHY overview：https://www.bluetooth.com/wp-content/uploads/2019/03/Bluetooth_5-FINAL.pdf
- Bluetooth LE Primer：https://www.bluetooth.com/bluetooth-le-primer/

## 4. 蓝牙优点

### 4.1 无线，穿戴体验好

没有网线，老人或被试移动更自由。

适合：

- 简单评估动作
- 居家使用
- 移动端查看状态
- 低频数据上传

### 4.2 低功耗

BLE 对电池友好。

适合：

- 长时间待机
- 电量状态同步
- 设备配置
- 低频传感数据

### 4.3 手机兼容性好

BLE 可以直接连手机 App。

适合：

- 设备绑定
- 查看电量
- 选择采集模式
- 开始/停止采集
- 查看低频摘要
- 设备固件升级，视方案而定

### 4.4 成本低

BLE 模块便宜，很多 MCU 自带 BLE。

适合：

- 量产设备配置链路
- 辅助通信链路
- 用户端交互

### 4.5 不需要布线

对养老、康复、现场演示比较友好。

但这只解决“方便”，不解决“高质量连续采集”。

## 5. 蓝牙缺点

### 5.1 吞吐余量不足

你们的原始数据至少约 1.6Mbps，float32 约 3.3Mbps。

BLE 即使用 2M PHY，实际稳定吞吐通常也会打折。不同手机、不同系统、不同 BLE 芯片表现差异很大。

风险：

- 掉帧
- 延迟抖动
- 缓存堆积
- 采集时长一长就不稳定
- 同一套设备换手机后表现不同

### 5.2 人体遮挡严重

压力衣穿在人身上，人体本身会吸收和遮挡 2.4GHz 信号。

风险：

- 转身后信号变差
- 手臂遮挡天线
- 相机/电脑位置变化影响连接
- 老人动作过程中链路质量波动

### 5.3 延迟和抖动不可控

BLE 可以传数据，但实时性不如有线 Ethernet。

对于回放对齐来说：

- 延迟不是最大问题。
- 抖动和丢帧更麻烦。

如果采集端没有完整本地缓存，蓝牙链路波动会直接损坏数据完整性。

### 5.4 多设备并发复杂

如果后续有：

- 多件压力衣
- 压力衣 + 手套
- 压力衣 + IMU
- 多个 BLE 外设

BLE 中央设备的连接数量、吞吐分配、调度都会变复杂。

### 5.5 调试成本高

BLE 问题经常不是代码逻辑问题，而是组合问题：

- 手机型号
- 操作系统版本
- BLE 芯片
- 连接参数
- MTU
- PHY
- 射频环境
- 天线位置

这会让 MVP 调试周期变长。

### 5.6 不适合大文件或原始流长期传输

BLE 可以传，但不是为高吞吐连续数据流设计的首选链路。

你们更应该让 BLE 做：

- 控制
- 状态
- 小数据
- 低频摘要

而不是做：

- 100Hz 全量压力流
- 原始数据长期采集
- 关键实验数据主链路

## 6. 蓝牙适合做什么

建议 BLE 用作辅助链路。

### 6.1 设备配置

手机 App 或采集软件通过 BLE 设置：

- 设备编号
- 被试编号
- 采样率
- Wi-Fi/Ethernet 参数，若有
- 校准模式
- 时间同步命令

### 6.2 状态监控

BLE 实时显示：

- 电量
- 设备温度
- 连接状态
- 当前采样率
- 是否正在采集
- 本地缓存剩余空间
- 错误码

### 6.3 开始/停止控制

BLE 可以发控制命令：

- start_capture
- stop_capture
- mark_event
- sync_flash
- calibrate

### 6.4 低频预览

BLE 可以传低频摘要，例如：

- 每秒 5-10 帧人体区域压力
- 不传 1024 点原始数据
- 只传胸、背、左腿、右腿、手臂等区域均值/峰值

示例：

```json
{
  "timestamp_ms": 123456,
  "region_chest": 12.4,
  "region_back": 8.1,
  "region_left_leg": 20.5,
  "region_right_leg": 22.0
}
```

### 6.5 本地存储后的数据下载

如果压力衣本地有 SD 卡或 Flash：

- 采集时数据先写本地。
- BLE 只做控制和状态。
- 采集后可通过 USB/Ethernet 导出。

不建议用 BLE 下载大文件，除非文件很小或用户能接受很慢。

## 7. 蓝牙不适合做什么

不建议 BLE 做：

- 全量 100Hz x 1024 点原始数据主传输
- 多小时连续高频采集
- 多设备同步采集
- 严格帧级同步
- 关键实验数据唯一链路
- 视频/音频同步主时间源

## 8. 可选架构

### 8.1 不推荐：纯 BLE 主链路

```text
压力衣 -> BLE -> 采集电脑/手机 -> 数据文件
```

优点：

- 无线。
- 成本低。

缺点：

- 吞吐吃紧。
- 稳定性风险高。
- 掉帧不好补。
- 回放对齐风险高。

结论：

> 不建议用于 MVP 主采集链路。

### 8.2 推荐：Ethernet 主链路 + BLE 辅助链路

```text
压力衣 -> Ethernet -> 采集电脑 -> 原始数据
压力衣 -> BLE -> 手机/采集电脑 -> 控制和状态
```

优点：

- 原始数据稳定。
- 控制体验方便。
- 方便后续做 App。
- 工程风险低。

缺点：

- 压力衣仍然有线。
- 需要同时维护两条通信链路。

结论：

> 推荐。

### 8.3 可选：本地存储 + BLE 控制

```text
压力衣 -> 本地 SD/Flash -> 原始数据
压力衣 -> BLE -> 手机/电脑 -> 控制、状态、低频预览
采集结束后 -> USB/Ethernet 导出原始数据
```

优点：

- 穿戴时无线。
- 原始数据不依赖 BLE 实时传输。
- 适合老人动作测试。

缺点：

- 不能实时完整回放。
- 需要本地存储可靠。
- 采集后导出流程多一步。

结论：

> 如果你们强烈希望无线穿戴，这是比“BLE 实时全量传输”更靠谱的方案。

## 9. 如果一定要用 BLE 传原始数据

如果必须尝试 BLE 传全量数据，需要降低风险：

### 9.1 数据压缩

可以考虑：

- uint16 定点数，不用 float32
- 差分编码
- 只传变化点
- 区域聚合
- 降采样

### 9.2 降低采样率或点位

例如：

- 100Hz 全量本地存储
- BLE 只传 10Hz 预览

或者：

- 1024 点全量本地存储
- BLE 只传 64 个区域聚合点

### 9.3 使用本地缓存

设备端必须有 ring buffer。

如果 BLE 暂时传不出去：

- 数据先缓存在设备本地。
- 后续补传。
- 不能直接丢帧。

### 9.4 做严格吞吐测试

必须测试：

- 10 分钟连续传输
- 30 分钟连续传输
- 不同距离
- 人体遮挡
- 转身
- 多手机型号
- Windows/macOS/iOS/Android 接收端

## 10. 蓝牙验收标准

如果 BLE 只做辅助链路，验收标准：

- 设备 5 秒内可被发现。
- 10 秒内可连接。
- 电量、采样状态、错误码可读取。
- start/stop 命令成功率 >= 99%。
- 状态刷新频率 >= 1Hz。
- 10 分钟连接不断开。
- 断开后能自动重连。

如果 BLE 做低频预览，验收标准：

- 预览频率 >= 5Hz。
- 延迟 < 500ms。
- 10 分钟预览不断流。
- 丢包或断流会明确提示。
- 不影响主链路原始数据采集。

如果 BLE 做全量原始数据传输，验收标准应更严格：

- 100Hz 连续采集 30 分钟。
- 单帧 1024 点完整。
- 缺失帧比例 < 1%。
- 时间戳连续。
- 接收端无长期缓存积压。
- 人体转身和遮挡时仍然稳定。
- 至少测试 3 种接收设备。

如果达不到这些指标，就不能把 BLE 作为主链路。

## 11. 与 Ethernet 对比

| 维度 | Bluetooth LE | Ethernet |
| --- | --- | --- |
| 是否无线 | 是 | 否 |
| 功耗 | 低 | 中等 |
| 全量数据稳定性 | 弱 | 强 |
| 吞吐余量 | 小 | 大 |
| 延迟抖动 | 较明显 | 小 |
| 多设备扩展 | 复杂 | 简单 |
| 手机连接 | 很方便 | 不方便 |
| 采集电脑连接 | 可行但不稳 | 很适合 |
| 调试复杂度 | 高 | 中 |
| 适合主数据链路 | 不推荐 | 推荐 |
| 适合控制链路 | 推荐 | 可用但笨重 |

## 12. 最终建议

### 12.1 MVP 推荐

```text
压力原始数据：Ethernet
设备状态和控制：BLE 可选
低频预览：BLE 可选
视频对齐：Ethernet 数据 + LED/蜂鸣同步事件
```

### 12.2 如果必须无线

不要用 BLE 实时传全量原始数据。

更推荐：

```text
压力衣本地存储全量数据
BLE 做控制和状态
采集结束后通过 USB/Ethernet 导出
```

### 12.3 产品路线

第一阶段：

- Ethernet 主链路
- 无 BLE 或 BLE 只做状态

第二阶段：

- BLE 控制
- 手机 App 配置
- 低频预览

第三阶段：

- 本地存储 + 无线控制
- 多设备管理
- 更完整移动端体验

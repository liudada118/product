# Ethernet 压力衣采集链路专项方案

> 项目：人体全身压力采集  
> 版本：v0.1  
> 日期：2026-05-17  
> 目标：把全身压力传感衣通过 Ethernet 接入采集软件，实现稳定、连续、可回放的数据采集。

## 1. Summary

本方案专门说明“压力衣通过 Ethernet 连接采集软件”的链路设计。

第一版建议采用：

> 压力衣控制器 -> Ethernet -> 千兆交换机/直连电脑 -> 采集软件 -> 本地数据文件 -> 回放工具

协议建议优先使用 `TCP 长连接`。第一版不需要 PTP、PoE、工业网管交换机或复杂硬同步。目标是先稳定采集 100Hz、1024 点位压力数据，并能与视频做回放对齐。

## 2. 为什么选 Ethernet

你们的数据量不大，但需要稳定。

按 `100Hz x 1024 点位` 估算：

| 数据格式 | 单帧大小 | 每秒数据量 | 网络占用 |
| --- | ---: | ---: | ---: |
| uint16，每点 2 字节 | 约 2KB | 约 200KB/s | 约 1.6Mbps |
| float32，每点 4 字节 | 约 4KB | 约 400KB/s | 约 3.2Mbps |
| 加帧头、时间戳、校验 | 约 2.2-4.5KB | < 500KB/s | < 5Mbps |

结论：

- `100Mbps Ethernet` 已经够用。
- `1GbE` 更推荐，主要是余量大、通用、稳定。
- 相比 BLE，Ethernet 带宽更足。
- 相比 Wi-Fi，Ethernet 延迟和丢包更可控。
- 相比 USB，Ethernet 更适合长线、隔离、多设备扩展。

## 3. 第一版目标

### 3.1 产品目标

让压力衣成为一个可被采集软件稳定连接的 Ethernet 设备。

采集软件可以：

- 发现或连接压力衣。
- 开始采集。
- 接收 100Hz 压力帧。
- 保存原始数据。
- 记录时间戳。
- 检测断线和异常。
- 为回放工具提供数据。

### 3.2 第一版不做

第一版暂不做：

- PoE 供电。
- PTP 精密时钟同步。
- 多件压力衣同步。
- 多相机硬件同步。
- 云端实时上传。
- 医疗级安全认证。
- 复杂设备管理平台。

原因：这些不是 MVP 成败关键，会显著增加硬件和软件复杂度。

## 4. 推荐硬件链路

### 4.1 最小链路

```text
压力传感阵列
  -> 采集前端
  -> 压力衣主控
  -> Ethernet 模块/网口
  -> 网线
  -> 采集电脑
  -> 采集软件
```

### 4.2 推荐链路

```text
压力衣
  -> RJ45
  -> Cat6 网线
  -> 千兆交换机
  -> 采集电脑
```

如果有相机：

```text
压力衣 ----\
相机 ------> 千兆交换机/采集电脑 -> 采集软件
同步器 ----/
```

第一版可以相机走 USB，压力衣走 Ethernet。这样更简单。

## 5. 硬件实现方案

### 5.1 方案 A：W5500 Ethernet 模块

链路：

```text
压力衣 MCU -> SPI -> W5500 -> RJ45
```

适合第一版。

优点：

- 上手快。
- 模块便宜。
- 资料多。
- 硬件 TCP/IP 协议栈，MCU 压力小。
- 足够传 100Hz x 1024 点位数据。

缺点：

- 灵活性不如 MCU 原生 Ethernet。
- SPI 和 buffer 需要调好。
- 长期产品化可能要重构。

建议：

> 如果你们之前没做过 Ethernet，先用 W5500 做样机。

### 5.2 方案 B：MCU 原生 Ethernet

链路：

```text
MCU Ethernet MAC -> PHY -> RJ45
```

适合第二版或量产版。

优点：

- 稳定性更好。
- 性能余量更大。
- 设备控制能力更强。
- 更适合长期产品。

缺点：

- PCB 和驱动复杂。
- 需要调 PHY、时钟、lwIP、EMC。
- 首版开发周期更长。

建议：

> 客户试点明确后，再做 MCU 原生 Ethernet 控制板。

### 5.3 方案 C：Linux 网关

链路：

```text
压力衣现有接口 -> Linux 小主机 -> Ethernet -> 采集电脑
```

适合快速验证。

优点：

- 软件开发最快。
- Python/C++ 都能做。
- 日志和调试方便。

缺点：

- 体积和功耗较大。
- 不适合最终穿戴产品。

建议：

> 如果你们现有压力衣已经能 USB/串口输出数据，但还没有 Ethernet，可以先用 Linux 网关过渡。

## 6. 推荐协议

### 6.1 第一版推荐 TCP

建议：

```text
TCP 长连接 + 每帧 length-prefix + 帧头 + payload + CRC
```

为什么不用 UDP：

- 第一版更关心数据完整性。
- UDP 要处理丢包、分包、乱序。
- 1024 点一帧可能超过普通 UDP 安全包长。
- TCP 对软件团队更友好。

### 6.2 连接角色

推荐：

```text
采集电脑 = TCP Server
压力衣 = TCP Client
```

原因：

- 采集电脑 IP 固定。
- 压力衣上电后主动连接。
- 多件设备也方便管理。
- 网络配置更简单。

也可以反过来：

```text
压力衣 = TCP Server
采集电脑 = TCP Client
```

这种方式适合单设备调试，但多设备管理会麻烦一些。

## 7. 数据帧格式

每一帧压力数据都必须可独立解释。

建议帧结构：

```text
[length: uint32]
[header]
[payload]
[crc32]
```

### 7.1 header 字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| magic | uint32 | 固定标识，防止错位解析 |
| protocol_version | uint16 | 协议版本 |
| device_id | uint32 | 压力衣编号 |
| session_id | uint32 | 本次采集编号 |
| frame_id | uint64 | 压力帧序号 |
| device_timestamp_us | uint64 | 设备端微秒时间戳 |
| sample_rate_hz | uint16 | 采样率，第一版 100 |
| sensor_count | uint16 | 点位数，第一版 1024 |
| value_type | uint8 | uint16 或 float32 |
| flags | uint16 | 是否同步事件、异常帧等 |
| payload_length | uint32 | payload 长度 |

### 7.2 payload 字段

payload 是 1024 个点位的压力值：

```text
sensor_0001
sensor_0002
...
sensor_1024
```

第一版建议内部传输用 `uint16` 或 `int32` 定点数，不建议直接用字符串 CSV 传输。

CSV 可以作为后处理导出格式。

## 8. 时间戳设计

第一版要保存三种时间：

| 时间 | 来源 | 用途 |
| --- | --- | --- |
| device_timestamp_us | 压力衣 | 记录设备内部采样时间 |
| host_receive_time_ns | 采集电脑 | 记录电脑收到帧的时间 |
| sync_event_time | 同步器/软件 | 对齐视频和压力数据 |

每帧保存：

```json
{
  "frame_id": 12345,
  "device_timestamp_us": 100234567,
  "host_receive_time_ns": 1779000000000000000
}
```

同步事件保存：

```json
{
  "event": "sync_flash_1",
  "host_time_ns": 1779000001234567890,
  "pressure_frame_id": 2468
}
```

这样后面才能做视频与压力帧对齐回放。

## 9. 同步策略

### 9.1 MVP 同步

第一版建议：

```text
软件时间戳 + LED 闪烁 + 蜂鸣器
```

流程：

1. 采集软件开始压力采集。
2. 相机开始录像。
3. 软件控制 LED 闪 3 次。
4. 软件控制蜂鸣器响 3 次。
5. 压力数据里记录 sync_event。
6. 后处理时用 LED 画面和 sync_event 对齐。

验收目标：

```text
视频和压力数据同步误差 < 500ms
```

### 9.2 后续同步

后续再考虑：

- PTP
- 相机硬件 trigger
- timecode
- ROS2 bag
- 同步脉冲线

触发条件：

- 客户要求更高同步精度。
- 同时采多台相机。
- 同时采多件压力衣。
- 要做算法训练而不只是回放展示。

## 10. IP 与网络配置

### 10.1 直连模式

```text
压力衣 <-> 采集电脑
```

推荐配置：

```text
采集电脑 IP：192.168.10.1
压力衣 IP：192.168.10.101
子网掩码：255.255.255.0
端口：5001
```

### 10.2 交换机模式

```text
压力衣 -> 千兆交换机 -> 采集电脑
```

推荐配置：

```text
采集电脑：192.168.10.1
压力衣 1：192.168.10.101
压力衣 2：192.168.10.102
相机 1：192.168.10.201
同步器：192.168.10.250
```

建议：

- 采集网络独立，不接办公室网络。
- 所有设备固定 IP。
- 使用 Cat6 网线。
- 使用千兆交换机。

## 11. 采集软件需求

采集软件必须支持：

- 监听压力衣连接。
- 显示设备在线状态。
- 显示采样率。
- 显示接收帧数。
- 显示丢帧/异常帧。
- 一键开始/停止采集。
- 写入原始数据。
- 写入采集日志。
- 写入 metadata。
- 断线提示。
- 重连后新建数据段，不覆盖旧数据。

建议软件模块：

```text
network_receiver
frame_parser
frame_validator
disk_writer
preview_buffer
capture_controller
```

## 12. 输出文件

每次采集输出一个样本目录：

```text
carry_box_s01_5kg_003/
  pressure_raw.bin
  pressure_index.csv
  pressure_clean.csv
  sync_events.json
  capture_log.jsonl
  metadata.json
```

如果同时采视频：

```text
  video.mp4
  video_metadata.json
```

`capture_log.jsonl` 需要记录：

- 连接时间
- 断线时间
- 开始采集时间
- 结束采集时间
- 总帧数
- CRC 错误数
- frame_id 跳变
- 实际采样率
- 设备状态

## 13. 验收标准

### 13.1 连接验收

- 采集电脑能发现或连接压力衣。
- 压力衣能稳定建立 TCP 连接。
- 连接后能返回设备信息。
- 断线后软件能提示。
- 重连后不会覆盖旧数据。

### 13.2 数据验收

- 连续采集 30 分钟不断线。
- 采样率稳定在 100Hz ± 5%。
- 每秒收到约 100 帧。
- 每帧包含 1024 点位。
- 每帧包含 frame_id。
- 每帧包含 device_timestamp_us。
- frame_id 单调递增。
- CRC 错误率 < 0.01%。
- 采集电脑无明显写盘积压。

### 13.3 回放验收

- 采集完成后能读取压力数据。
- 能按时间轴播放压力帧。
- 能与视频做对齐。
- 暂停时能显示当前压力帧编号。
- 拖动时间轴后能跳到对应压力帧。

### 13.4 异常验收

- 拔掉网线，软件 3 秒内提示断线。
- 重新插回网线，设备可重新连接。
- 设备重启后软件能识别新 session 或新 segment。
- 数据异常会写入 quality report。

## 14. 测试场景

### 测试 1：基础连接

目标：验证压力衣能通过 Ethernet 与采集软件建立连接。

步骤：

1. 压力衣接入交换机。
2. 采集电脑启动软件。
3. 压力衣上电。
4. 观察软件设备列表。

预期：

- 软件显示设备在线。
- 能看到 device_id、firmware、sample_rate、sensor_count。

### 测试 2：连续采集

目标：验证 30 分钟稳定性。

步骤：

1. 开始采集。
2. 连续运行 30 分钟。
3. 停止采集。
4. 查看日志和文件。

预期：

- 总帧数约 180000 帧。
- 无断线。
- 实际采样率在 95-105Hz。
- CRC 错误率 < 0.01%。

### 测试 3：动作样本采集

目标：验证一个完整动作样本可保存。

步骤：

1. 创建样本 `carry_box_s01_5kg_001`。
2. 开始采集。
3. 被试完成搬箱动作。
4. 停止采集。
5. 打开输出目录。

预期：

- 有 pressure_raw.bin。
- 有 metadata.json。
- 有 capture_log.jsonl。
- 数据能被回放工具读取。

### 测试 4：断线恢复

目标：验证异常处理。

步骤：

1. 开始采集。
2. 采集中拔掉网线。
3. 等待 5 秒。
4. 插回网线。
5. 观察软件状态。

预期：

- 软件提示断线。
- 日志记录断线时间。
- 重连后新建 segment。
- 旧数据不丢失、不覆盖。

## 15. 推荐开发顺序

### 第 1 步：先做 PC 模拟器

写一个程序模拟压力衣，通过 TCP 发 100Hz、1024 点假数据。

目的：

- 先把协议、采集软件、写盘、回放打通。
- 不等硬件。

### 第 2 步：接真实压力衣

用 W5500 或 Linux 网关把真实压力数据发出来。

目的：

- 验证真实帧率。
- 验证时间戳。
- 验证断线重连。

### 第 3 步：做采集样本

采 10 条搬箱样本。

目的：

- 验证数据文件结构。
- 验证回放。
- 验证质量报告。

### 第 4 步：再决定是否产品化硬件

如果客户反馈确认，再做 MCU 原生 Ethernet 控制板。

第一版建议采购：

| 物品 | 数量 | 说明 |
| --- | ---: | --- |
| 千兆交换机 | 1 | 普通非网管即可 |
| Cat6 网线 | 3-5 | 1m、3m、5m 各准备 |
| USB 千兆网卡 | 1 | 笔记本无网口时使用 |
| W5500 模块 | 2-3 | 硬件 MVP 使用 |
| ESP32/STM32 开发板 | 1-2 | 视现有主控而定 |
| RJ45 转接板 | 若干 | 调试使用 |

暂不采购：

- PoE 交换机
- PTP grandmaster
- 工业网管交换机
- 万兆网卡
- 光纤模块

## 17. 最终建议

第一版走这个路线：

```text
Ethernet 物理链路：压力衣 -> 网线 -> 千兆交换机/采集电脑
传输协议：TCP 长连接
数据格式：length + header + payload + crc32
同步方式：device timestamp + host timestamp + LED/蜂鸣同步事件
验收目标：30 分钟稳定采集，100Hz，1024 点，回放可对齐视频
```

这个方案成本低、风险小、开发路径清晰，足够支持“搬运箱子数据样例包”的第一版采集需求。

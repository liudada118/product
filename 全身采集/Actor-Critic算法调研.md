# Actor-Critic 算法调研

文档日期：2026-05-12  
文档对象：全身采集 / 全身传感衣 / 机器人运动数据采集方向  
文档类型：技术调研与产品落地判断  

## 1. 调研结论

| 结论 | 说明 |
|---|---|
| Actor-Critic 不是单一算法，而是一类强化学习框架 | Actor 负责输出动作策略，Critic 负责评估当前策略或动作价值。它适合连续动作、高维控制、机器人运动控制等问题。 |
| 对“全身采集”最相关的不是纯 Actor-Critic，而是 PPO、SAC、DDPG/TD3 这类衍生算法 | 全身运动、姿态控制、机器人遥操作和仿真训练通常需要连续动作空间、稳定训练和仿真到现实迁移，PPO/SAC 更常用。 |
| 如果当前产品重点是“采集数据”，Actor-Critic 不是 V1 必做功能 | V1 更应该先做好高质量数据采集、标定、同步、回放、导出、SDK。Actor-Critic 更适合作为后续“用采集数据训练控制策略”的算法模块。 |
| 如果产品面向“机器人模仿学习/强化学习训练平台”，Actor-Critic 值得提前预留数据接口 | 需要保留状态、动作、奖励、终止条件、时间戳、任务标签、人体动作标签、机器人映射动作等字段。 |
| 推荐路线：先做数据闭环，再接 PPO/SAC | 第一阶段采集与回放；第二阶段离线模仿学习；第三阶段在仿真环境中用 PPO/SAC 做策略微调；第四阶段考虑真机安全部署。 |

## 2. Actor-Critic 是什么

Actor-Critic 是强化学习中的一类策略优化方法，由两个角色组成：

| 角色 | 作用 | 类比 |
|---|---|---|
| Actor | 学习策略 `pi(a|s)`，决定在状态 `s` 下采取什么动作 `a` | 控制器 / 决策器 |
| Critic | 学习价值函数 `V(s)`、`Q(s,a)` 或优势函数 `A(s,a)`，评估 Actor 的动作好坏 | 评估器 / 教练 |

核心思想是：Actor 直接改进策略，Critic 提供更低方差、更稳定的学习信号。相比只学习价值函数的方法，Actor-Critic 更适合连续动作；相比纯策略梯度方法，它通常训练效率更高。

## 3. 典型算法谱系

| 算法 | 类型 | 关键特征 | 适合场景 | 对全身采集的启发 |
|---|---|---|---|---|
| 原始 Actor-Critic | 在线策略学习 | Actor 更新策略，Critic 估计价值 | 教学与基础研究 | 理解框架即可，工程上通常不直接用原始版本。 |
| A3C | On-policy | 多线程异步采样，提高训练效率 | 仿真任务、离散/连续控制 | 说明并行采样对训练很重要，全身采集平台可考虑多任务、多被试并行采集。 |
| DDPG | Off-policy | 面向连续动作，确定性策略，经验回放 | 机器人连续控制 | 对机器人关节控制有参考，但训练稳定性弱于后续 TD3/SAC。 |
| PPO | On-policy | 限制策略更新幅度，工程稳定性好 | 机器人仿真、运动控制、游戏 AI | 如果要做全身动作策略训练，PPO 是优先候选。 |
| SAC | Off-policy | 最大熵强化学习，鼓励探索，样本效率较高 | 机器人控制、连续动作、仿真训练 | 如果采集成本高、希望复用经验数据，SAC 更有吸引力。 |
| TD3 | Off-policy | DDPG 改进版，双 Critic，延迟策略更新 | 连续控制 | 可作为 SAC 的替代方案，但产品化资料通常不如 PPO/SAC 易讲。 |

## 4. 为什么它和全身采集相关

全身采集产品本身采的是人体动作、姿态、压力、触觉、生理或交互数据。Actor-Critic 真正相关的地方，是把这些数据变成机器人或虚拟人的可执行策略。

### 4.1 数据采集层

全身采集系统应尽量记录可供强化学习使用的数据字段：

| 数据字段 | 说明 |
|---|---|
| `timestamp` | 统一时间戳，保证多传感器同步。 |
| `body_pose` | 人体骨架、关节角、IMU 姿态、位置数据。 |
| `contact_pressure` | 身体接触、足底压力、局部压力分布。 |
| `action_label` | 走、蹲、转身、抓取、搬运等动作标签。 |
| `task_context` | 当前任务、目标物、环境、工具、约束条件。 |
| `robot_mapping` | 人体动作到机器人关节或末端执行器的映射结果。 |
| `reward_signal` | 成功/失败、稳定性、能耗、轨迹误差、接触安全等奖励线索。 |
| `done_flag` | 任务结束、跌倒、碰撞、超限等终止条件。 |

如果 V1 没有这些字段，后续再做强化学习模块会很难补齐数据闭环。

### 4.2 训练层

Actor-Critic 可以用于三类训练：

| 训练方式 | 描述 | 适合度 |
|---|---|---|
| 直接强化学习 | 在仿真环境中让策略自己试错学习 | 适合仿真，不适合直接真机高风险试错。 |
| 模仿学习 + 强化学习微调 | 先用人体采集数据训练初始策略，再用 PPO/SAC 微调 | 最适合全身采集产品讲故事。 |
| 人在回路强化学习 | 人体动作、人工评分或纠偏作为训练信号 | 适合机器人遥操作、康复训练、复杂任务示教。 |

### 4.3 应用层

| 应用 | Actor-Critic 的作用 | 产品价值 |
|---|---|---|
| 人形机器人动作学习 | 学习走路、转身、搬运、平衡控制 | 把全身采集从“记录工具”升级成“训练数据入口”。 |
| 康复训练 | 根据患者动作质量调整辅助策略 | 可形成个性化训练与评估闭环。 |
| VR/仿真训练 | 训练虚拟角色或交互智能体 | 适合全身传感衣与仿真平台结合。 |
| 远程操作 | 将人类动作映射到机器人，并通过 Critic 评估安全和任务效果 | 可做“示教 + 自动策略优化”。 |

## 5. PPO、SAC、DDPG 的选型建议

| 维度 | PPO | SAC | DDPG/TD3 |
|---|---|---|---|
| 训练稳定性 | 高 | 高 | DDPG 中等，TD3 较好 |
| 样本效率 | 中等 | 较高 | 较高 |
| 工程上手难度 | 较低 | 中等 | 中等 |
| 是否适合连续动作 | 适合 | 很适合 | 很适合 |
| 是否适合机器人仿真 | 很适合 | 很适合 | 适合 |
| 对采集数据复用 | 一般 | 较好 | 较好 |
| 推荐优先级 | P0 | P0/P1 | P1 |

建议：

1. 产品展示和仿真控制优先选 PPO，因为稳定、资料多、客户容易理解。
2. 如果强调“采集成本高、样本要复用”，优先研究 SAC。
3. DDPG 可以作为历史技术脉络讲清楚，但不建议作为新产品主推算法。
4. 如果后续要做机器人真机控制，应优先在仿真中训练，再做安全约束、动作限幅和人工接管。

## 6. 对全身采集产品的落地建议

### 6.1 V1 不建议直接做强化学习训练平台

原因：

1. 强化学习依赖仿真环境、奖励函数、动作空间、状态空间、终止条件，不是只有采集硬件就能完成。
2. 真机试错成本高，容易带来安全风险。
3. 客户在早期更关心采集精度、延迟、稳定性、标定、导出、SDK，而不是算法名词。

V1 更合理的目标是：让采集数据足够干净、完整、可回放、可训练。

### 6.2 V1 应该预留的能力

| 能力 | 优先级 | 说明 |
|---|---|---|
| 多传感器时间同步 | P0 | 强化学习训练最怕状态和动作错位。 |
| 原始数据导出 | P0 | CSV/JSON/ROS bag/Parquet 至少预留一种结构化导出。 |
| 动作标签体系 | P0 | 没有标签，后续模仿学习和任务分段成本很高。 |
| 回放与关键帧标注 | P0 | 便于清洗数据、标注成功/失败和动作阶段。 |
| SDK/API | P0 | Python 优先，后续对接 Gymnasium、Isaac Sim、MuJoCo、ROS。 |
| 奖励字段预留 | P1 | 可以先不计算奖励，但数据结构要能保存奖励。 |
| 机器人映射层 | P1 | 人体骨架到机器人关节、末端位姿或速度控制的映射。 |
| 仿真接口 | P2 | 后续对接强化学习环境。 |

### 6.3 后续算法模块可以这样包装

模块名称建议：`采集数据驱动的机器人策略训练模块`

核心流程：

1. 采集人体全身动作和接触数据。
2. 清洗、分段、标注动作和任务结果。
3. 用模仿学习训练初始策略。
4. 在仿真环境中用 PPO/SAC 微调。
5. 通过安全约束和人工接管机制迁移到机器人。

对外不要只讲“Actor-Critic”，而要讲“让机器人从人体动作数据中学习可执行策略”。

## 7. 风险与限制

| 风险 | 说明 | 应对建议 |
|---|---|---|
| 奖励函数难定义 | 很多全身动作不是简单成功/失败，奖励设计会影响策略质量。 | 从任务型场景切入，例如搬运、站起、保持平衡、避障。 |
| 仿真到现实差距 | 仿真训练策略到真实机器人可能失效。 | 使用域随机化、动作限幅、在线校正和安全控制。 |
| 数据维度高 | 全身姿态、压力、触觉数据维度很高。 | 做特征压缩、关键点选择、分层控制。 |
| 人体到机器人映射不唯一 | 人和机器人骨架不同，动作不能直接复制。 | 增加 retargeting 层，把人体动作映射为机器人约束下的目标。 |
| 真机探索危险 | 强化学习试错可能导致碰撞、跌倒或设备损坏。 | 强化学习放在仿真中，真机阶段加入安全监控和人工接管。 |

## 8. 推荐下一步验证

| 阶段 | 验证问题 | 交付物 |
|---|---|---|
| 1 | 全身采集数据能否稳定导出为训练数据集？ | 数据字段规范、样例数据、回放工具。 |
| 2 | 采集数据能否训练一个简单模仿学习模型？ | 基于人体动作的预测/复现 demo。 |
| 3 | 能否接入仿真环境做 PPO/SAC 微调？ | Gymnasium/Isaac Sim/MuJoCo 环境接口。 |
| 4 | 策略能否迁移到机器人或虚拟人？ | 安全受限的动作控制 demo。 |

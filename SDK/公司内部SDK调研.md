# 公司内部 SDK 调研

> 版本：v0.1  
> 日期：2026-05-16  
> 目标：调研公司内部 SDK 的定位、架构、组件范围、Agent 使用方式，以及算法能力本地部署和服务器部署的取舍。

## 1. 一句话结论

建议把公司内部 SDK 定位为：

> 面向内部业务软件快速搭建的能力层，把常用业务能力、渲染 UI、数据处理、算法调用和工程规范封装成稳定接口，让懂业务的伙伴可以通过 Agent 快速拼出可用软件，让不懂业务的伙伴可以先通过 PRD 把需求说清楚，再由 Agent 和研发落地。

第一版不要做成“大而全平台”，而应先做成一套小而稳定的内部工具包：

- 业务组件库：封装公司常用业务对象、流程和页面模块。
- UI 渲染库：封装常用图表、表格、表单、回放、热力图、报告页面。
- 算法调用层：统一本地算法和服务器算法的调用方式。
- Agent 友好文档：让 Agent 能读懂组件能力、参数、示例和组合规则。
- PRD 模板：帮助不懂业务或不懂技术的伙伴把需求转成可执行描述。

算法部署建议采用“服务器优先，本地可选”的混合方案：

> 核心算法、商业价值高的算法、需要持续更新的算法优先放服务器；离线演示、弱网场景、低延迟采集链路可保留本地插件模式。

## 2. 背景与目标

公司内部已经积累了一些业务方向，例如压力传感、人体采集、评估系统、数据回放、报告生成、设备数据接入等。后续如果每次做新软件都从零开始，会出现几个问题：

- 业务逻辑反复重写，结果口径容易不一致。
- UI 页面重复开发，研发效率低。
- 算法接入方式分散，本地环境搭建和版本管理复杂。
- Agent 即使能写代码，也缺少对公司业务组件和数据结构的稳定理解。
- 非业务人员提需求时容易只描述想法，缺少边界、流程和验收标准。

内部 SDK 的目标不是替代产品经理、研发或算法工程师，而是把重复能力沉淀下来，让需求到软件原型的路径更短、更稳定。

## 3. 目标用户

### 3.1 懂业务的伙伴

这类人知道业务流程、客户场景和数据含义，但不一定想从零写软件。

他们的理想使用方式是：

> 直接向 Agent 描述需求，Agent 根据 SDK 文档选择组件、拼接流程、生成页面和数据逻辑。

典型例子：

- “做一个压力数据采集后的回放工具，左边视频，右边人体热力图，下面显示关键区域曲线。”
- “做一个老人肌少症评估页面，录入握力、步速、SARC-F，自动生成风险报告。”
- “做一个搬运箱子样例包浏览器，可以筛选动作阶段、重量和受试者。”

### 3.2 不懂业务但需要提需求的伙伴

这类人可能只知道目标结果，不清楚业务字段、流程和算法边界。

建议提供一套 PRD 模板，让他们先回答：

- 这个软件服务谁？
- 用户要完成什么任务？
- 输入数据从哪里来？
- 输出结果给谁看？
- 哪些环节必须自动化，哪些可以人工确认？
- 哪些算法结果只是辅助判断，哪些会影响业务决策？
- 验收时看哪些指标或页面？

PRD 的作用不是写得很正式，而是把需求变成 Agent 和研发都能执行的结构化描述。

## 4. SDK 应解决的问题

| 问题 | 当前风险 | SDK 应提供的能力 |
| --- | --- | --- |
| 页面重复开发 | 每个项目都重写表格、表单、图表、报告 | 提供标准 UI 组件和页面模板 |
| 业务口径不一致 | 同一指标在不同软件中算法不同 | 提供统一业务模型和计算函数 |
| 算法接入复杂 | 环境、依赖、版本、模型路径难管理 | 提供统一算法调用接口 |
| Agent 不懂内部业务 | 生成代码可运行但不符合业务 | 提供 Agent 可读的组件说明、示例和约束 |
| 非技术需求难落地 | 需求描述发散，缺输入输出和验收 | 提供 PRD 模板和需求检查清单 |
| 数据格式分散 | 不同项目字段命名不同 | 提供标准数据结构和转换器 |

## 5. 第一版 SDK 范围

第一版建议只做内部高频、低争议、容易复用的能力。

### 5.1 业务基础包

建议封装：

- 项目、受试者、采集任务、样本、设备、传感器、评估报告等基础对象。
- 压力数据、视频数据、动作阶段标签、人体区域映射等数据结构。
- 常用单位、枚举、字段校验、时间戳同步规则。
- 数据导入、导出、清洗、格式转换工具。

不建议第一版封装：

- 复杂权限系统。
- 完整客户管理系统。
- 完整数据中台。
- 所有业务方向的算法细节。

### 5.2 UI 渲染包

建议封装：

- 标准表单：基础信息录入、采集任务配置、评估指标录入。
- 标准表格：样本列表、设备列表、评估记录、质量检查记录。
- 数据可视化：压力曲线、区域对比、时间轴、阶段标签。
- 专业视图：视频 + 压力热力图同步回放、人体区域图、报告预览。
- 报告页面：评估报告、质量报告、样例包说明、客户演示报告。

UI 组件要重点解决两个问题：

1. 让 Agent 不需要重新设计常见页面。
2. 让不同项目的软件看起来像同一家公司做出来的产品。

### 5.3 算法调用包

建议封装统一接口，而不是让业务代码直接依赖某个本地脚本或服务器接口。

示例能力：

- `runAssessment(input)`：运行评估逻辑。
- `detectMotionPhase(input)`：识别动作阶段。
- `syncVideoAndSensor(input)`：视频与传感器数据同步。
- `generateReport(input)`：生成报告建议和解释文本。
- `calculatePressureFeatures(input)`：计算压力特征。

业务软件只关心输入和输出，不关心算法部署在本地还是服务器。

## 6. Agent 拼组件的工作流

建议把 SDK 设计成 Agent 友好，而不是只给人看。

### 6.1 懂业务伙伴的链路

1. 伙伴用自然语言描述软件目标和关键页面。
2. Agent 读取 SDK 文档，选择业务对象和 UI 组件。
3. Agent 生成软件结构、页面和数据流。
4. 人检查业务逻辑和页面。
5. Agent 根据反馈迭代。

关键要求：

- 每个组件都要有用途、参数、返回值、示例和禁用场景。
- 每个业务对象都要有字段含义、单位、是否必填和示例数据。
- 每个页面模板都要有适用场景和组合方式。

### 6.2 不懂业务伙伴的链路

1. 先填写 PRD 模板。
2. Agent 根据 PRD 反问缺失信息。
3. PRD 达到可执行程度后，再进入 SDK 拼装。
4. 业务负责人确认关键口径。
5. 研发或 Agent 完成软件原型。

关键要求：

- PRD 模板要逼近输入、处理、输出、验收。
- 对算法结果要区分“辅助建议”和“自动决策”。
- 对医疗、评估、风险判断类软件要保留人工确认边界。

## 7. 算法部署方案对比

算法能力大致有两种部署方式：本地部署和服务器部署。实际建议不要二选一，而是统一接口后按场景选择。

### 7.1 本地部署

本地部署指算法模型、运行环境、依赖库和数据处理程序都在用户电脑或内网机器上运行。

#### 优点

- **低延迟**：采集、回放、实时检测场景更容易做到快速响应。
- **可离线**：没有网络时也能演示或运行。
- **数据不出本机**：敏感原始数据可以不上传服务器。
- **适合设备强绑定链路**：例如硬件采集软件、串口/蓝牙/USB 设备、实时传感器流。
- **适合交付给封闭内网客户**：有些客户不允许数据出内网。

#### 缺点

- **环境搭建麻烦**：Python、CUDA、驱动、模型文件、系统版本都可能出问题。
- **升级困难**：每台机器都要更新，版本不一致会导致结果不一致。
- **排障成本高**：客户现场问题可能来自硬件、系统、依赖、权限、显卡或模型路径。
- **破解和泄露风险更高**：模型文件、算法代码、授权逻辑在本地，更容易被复制、逆向或绕过。
- **算力不可控**：不同电脑性能差异大，算法运行速度和稳定性不一致。
- **日志和质量监控弱**：很难统一统计算法错误、耗时、输入分布和结果质量。

#### 破解风险说明

本地部署无法彻底避免破解，只能提高成本。

常见风险：

- 模型文件被直接复制。
- Python 或前端代码被反编译/阅读。
- 授权校验被绕过。
- 接口参数和算法流程被逆向。
- 离线授权文件被复用。

可降低风险的方式：

- 核心算法不下发，只下发轻量推理或预处理能力。
- 使用许可证、设备指纹、到期校验、加密模型文件。
- 把关键商业规则放服务器，本地只做非核心计算。
- 对客户交付采用合同、授权和审计机制约束。

但要有清醒判断：

> 只要核心模型和算法完整落到客户机器上，就不能把“防破解”当成绝对安全能力。

### 7.2 服务器部署

服务器部署指算法运行在公司服务器、私有云或客户内网服务器上，客户端通过 API 调用。

#### 优点

- **保护核心算法**：模型和算法代码不直接交付给终端用户。
- **统一升级**：服务器更新一次，所有客户端都使用新版本。
- **结果口径一致**：不同项目调用同一版本算法，减少分叉。
- **便于监控**：可以记录请求量、耗时、错误、版本、结果分布。
- **便于权限控制**：账号、额度、接口权限、客户隔离都更好管理。
- **便于商业化**：可以按调用量、项目、账号、并发、套餐收费。
- **适合大模型或重算法**：客户端不用配置复杂环境，也不用高性能显卡。

#### 缺点

- **依赖网络**：弱网、断网或客户内网限制会影响使用。
- **服务器成本**：需要承担 GPU/CPU、存储、带宽、运维和安全成本。
- **响应延迟**：实时采集和低延迟交互场景可能受影响。
- **数据合规压力更大**：如果上传人体、医疗、客户业务数据，需要明确授权、脱敏、存储和删除策略。
- **高可用要求更高**：服务挂了会影响多个项目。
- **客户私有化诉求**：一些客户可能要求部署在他们自己的内网或私有云。

### 7.3 私有化服务器部署

还有一种折中方案：算法不放在普通用户电脑上，而是部署在客户内网服务器或边缘服务器。

适合场景：

- 客户不允许数据出内网。
- 算法比较核心，不希望散落在多台终端电脑。
- 多台采集设备需要统一调用同一个算法服务。
- 客户愿意为私有化部署和维护付费。

优点：

- 数据仍在客户内网。
- 算法不直接暴露给每台终端。
- 版本更好管理。
- 比纯本地更容易监控和维护。

缺点：

- 部署和运维成本仍然较高。
- 客户环境差异大。
- 需要容器化、授权、日志、升级机制。

## 8. 推荐部署策略

建议采用分层部署：

| 算法类型 | 推荐部署 | 原因 |
| --- | --- | --- |
| 核心商业算法 | 公司服务器或客户私有化服务器 | 保护算法，便于升级和计费 |
| 高频实时采集预处理 | 本地 | 低延迟，依赖设备链路 |
| 非核心工具函数 | SDK 本地包 | 简单稳定，无需频繁请求服务器 |
| 大模型、多模态分析 | 服务器 | 环境复杂，算力要求高 |
| 离线演示能力 | 本地轻量版 | 保障无网演示 |
| 医疗/评估报告建议 | 服务器优先 | 统一口径，便于审计和版本控制 |

推荐第一版技术原则：

1. SDK 只暴露统一算法接口，不让业务软件直接绑定具体部署方式。
2. 算法服务返回结果必须带版本号、输入摘要、置信度或质量标记。
3. 本地算法只承担必要的设备接入、预处理和离线演示能力。
4. 核心算法、模型和规则优先放服务器。
5. 对客户交付时保留三种模式：公司云服务、客户私有化服务、本地轻量版。

## 9. SDK 架构建议

建议从逻辑上拆成五层：

```text
应用层
  业务软件、Demo、客户交付工具、内部运营工具

Agent 编排层
  需求理解、组件选择、页面生成、PRD 检查、代码生成规则

SDK 能力层
  业务对象、UI 组件、数据转换、算法接口、报告模板

服务层
  算法服务、文件服务、认证授权、日志监控、版本管理

数据层
  原始传感器数据、视频、标签、评估记录、报告、客户项目数据
```

第一版 SDK 不一定要一次实现所有层，但文档和接口设计要按这个方向预留。

## 10. Web 版与 Electron 版选型调研

内部 SDK 后续会服务两类软件：

- 面向内部协作、管理、报告、审核、数据浏览的软件。
- 面向采集设备、本地文件、离线演示、实时回放的软件。

这两类软件对运行环境的要求不同，因此需要先判断 Web 版和 Electron 版各自适合什么场景。

### 10.1 基本定义

#### Web 版

Web 版指软件运行在浏览器里，通过网址访问。它可以是普通网页应用，也可以进一步做成 PWA，让用户像安装应用一样使用，并支持一定程度的离线缓存和本地存储。

典型形态：

- 内部管理后台。
- 数据浏览平台。
- 报告生成和审核系统。
- 轻量标注工具。
- 客户在线 Demo。
- 私有化部署到客户内网的 Web 系统。

#### Electron 版

Electron 版指用 Web 技术写界面，但打包成桌面应用。Electron 官方文档说明，它通过嵌入 Chromium 和 Node.js，让同一套 JavaScript/HTML/CSS 代码可以构建 Windows、macOS、Linux 桌面应用。

典型形态：

- 本地采集软件。
- 本地回放工具。
- 设备调试工具。
- 离线演示软件。
- 大文件批处理工具。
- 需要访问本地算法、文件系统、串口、USB、蓝牙或本地服务的软件。

参考：

- [Electron 官方介绍](https://www.electronjs.org/docs/latest/)
- [Electron Why Electron](https://www.electronjs.org/docs/latest/why-electron)
- [MDN PWA 介绍](https://developer.mozilla.org/docs/Web/Progressive_web_apps/Guides/What_is_a_progressive_web_app)

### 10.2 核心区别

| 维度 | Web 版 | Electron 版 |
| --- | --- | --- |
| 访问方式 | 浏览器打开网址 | 安装桌面客户端 |
| 更新方式 | 服务端更新后用户自动使用新版 | 需要客户端升级或自动更新机制 |
| 部署成本 | 低，适合快速迭代和多人访问 | 中高，需要打包、签名、分发、升级 |
| 跨平台 | 浏览器天然跨平台，但受浏览器能力限制 | 可打包 Windows/macOS/Linux，但要处理平台差异 |
| 本地文件访问 | 受浏览器权限限制 | 能更深访问本地文件系统 |
| 硬件设备访问 | 受浏览器 API 和权限限制 | 更适合串口、USB、蓝牙、驱动、本地服务 |
| 离线能力 | PWA 可做部分离线，但复杂任务受限 | 更适合完整离线使用 |
| 算法本地运行 | 不适合重算法本地运行，通常调用服务器 | 可调用本地模型、Python 服务或二进制程序 |
| 安全边界 | 浏览器沙箱更清晰 | 权限更大，安全设计要求更高 |
| 性能和资源 | 依赖浏览器，轻量 | 包体和内存更大 |
| 数据协作 | 天然适合多人协作和集中数据 | 更适合单机或局域网场景 |
| Agent 生成难度 | 更容易标准化 | 需要额外处理桌面能力和桥接层 |

### 10.3 Web 版优点

Web 版最适合做内部 SDK 的默认交付形态。

优点：

- **上线快**：部署到服务器后，所有人访问同一个地址。
- **更新简单**：修复问题或新增组件后，用户刷新页面即可使用。
- **协作方便**：账号、权限、项目、数据、报告天然适合集中管理。
- **维护成本低**：不用处理安装包、操作系统差异、自动更新、证书签名。
- **更适合 Agent 生成**：页面、路由、表单、图表、接口调用都更标准。
- **更适合内部 SDK 沉淀**：UI 组件、业务对象、接口客户端可以在多个 Web 项目复用。
- **安全边界相对清晰**：浏览器默认沙箱限制了网页直接访问系统资源。

适合的公司场景：

- 项目管理、客户管理、样本管理。
- 数据集浏览、筛选、搜索、质量检查。
- 报告生成、报告审核、报告分享。
- 评估系统、问卷录入、指标计算。
- 业务 Demo 和销售演示。
- Agent 根据 PRD 快速生成内部工具。

### 10.4 Web 版缺点

Web 版的问题主要集中在本地能力不足。

缺点：

- **硬件访问受限**：串口、USB、蓝牙、摄像头、传感器等能力受浏览器支持和权限限制。
- **本地文件能力受限**：虽然现代浏览器已有部分文件访问能力，但仍不如桌面应用自由。
- **离线能力有限**：PWA 可以缓存页面和部分数据，但复杂离线流程、超大文件处理、长时间后台任务仍不稳定。
- **低延迟采集不理想**：实时采集、实时同步、实时算法推理可能受浏览器和网络限制。
- **依赖服务端**：如果核心能力都在服务器，断网时可用性会下降。
- **客户内网部署仍要运维**：私有化 Web 需要服务器、域名或内网地址、数据库、备份和权限体系。

因此，Web 版不适合作为唯一形态覆盖所有采集和设备场景。

### 10.5 Electron 版优点

Electron 版最适合做“带本地能力的桌面工具”。

优点：

- **本地能力强**：可以访问本地文件、调用本地服务、执行本地脚本、连接设备。
- **离线体验好**：可以在无网环境完成采集、回放、演示和本地处理。
- **适合硬件链路**：串口、USB、蓝牙、驱动、SDK、采集卡等通常更适合桌面端集成。
- **适合大文件处理**：本地视频、传感器数据、日志、模型文件可以直接处理，减少上传等待。
- **适合本地算法**：可以调用本地 Python、C++、ONNX、CUDA 或独立推理服务。
- **用户体验像桌面软件**：可做菜单、托盘、文件关联、自动启动、本地通知等能力。

适合的公司场景：

- 压力传感衣采集软件。
- 设备调试和标定工具。
- 视频与压力数据本地同步回放。
- 客户现场无网演示工具。
- 本地大文件批量转换、清洗、上传。
- 本地轻量算法推理和结果预览。

### 10.6 Electron 版缺点

Electron 的问题不是不能做，而是工程治理成本更高。

缺点：

- **包体较大**：Electron 会携带 Chromium 和运行时，安装包通常明显大于普通 Web。
- **资源占用更高**：内存和启动成本通常高于原生轻量客户端。
- **分发复杂**：Windows、macOS、Linux 都要分别打包、签名、发布和测试。
- **升级复杂**：需要自动更新机制，否则客户容易停留在旧版本。
- **安全风险更高**：Electron 同时涉及 Chromium、Node.js、NPM 依赖和应用自身代码。官方安全文档也强调，应用安全取决于框架基础、依赖和自身代码的整体安全。
- **权限边界更敏感**：如果渲染进程暴露了 Node.js 或本地能力，前端漏洞可能升级成本地执行风险。
- **破解风险仍存在**：Electron 不是算法保护方案，本地代码、模型、配置和授权逻辑仍可能被提取或逆向。
- **Agent 生成需要更多约束**：Agent 不能随意把系统能力暴露给页面，需要通过受控桥接层访问。

参考：

- [Electron 安全文档](https://www.electronjs.org/docs/latest/tutorial/security)
- [Electron 进程沙箱文档](https://www.electronjs.org/docs/latest/tutorial/sandbox)

### 10.7 PWA 作为中间方案

PWA 可以把 Web 版向桌面体验推进一步。

它可以提供：

- 安装到桌面或启动器。
- 离线缓存页面资源。
- 使用 Service Worker 处理缓存和后台能力。
- 使用 IndexedDB、Cache API 等保存本地数据。
- 在部分浏览器中支持文件处理、推送通知等能力。

但 PWA 仍然不是 Electron 的完全替代品。

适合 PWA 的场景：

- 报告查看器。
- 轻量数据浏览器。
- 可离线查看的客户 Demo。
- 低复杂度表单和评估工具。

不适合 PWA 的场景：

- 强依赖硬件设备。
- 需要长期后台采集。
- 需要调用本地 Python/C++ 算法。
- 需要复杂本地文件系统管理。
- 需要严格可控的桌面运行环境。

### 10.8 对算法部署的影响

Web 版和 Electron 版对算法部署影响很大。

| 算法部署方式 | Web 版适配 | Electron 版适配 | 说明 |
| --- | --- | --- | --- |
| 公司服务器算法 API | 强 | 强 | Web 和 Electron 都可以调用，是最推荐的核心算法方式 |
| 客户私有化算法服务 | 强 | 强 | 两者都可以访问客户内网服务 |
| 本地轻量算法 | 弱 | 强 | Web 通常受限，Electron 更适合 |
| 本地重算法/GPU 推理 | 弱 | 中 | Electron 可调用，但安装和驱动成本高 |
| 浏览器端 WASM 推理 | 中 | 中 | 可做轻量推理，但大模型和复杂依赖不适合 |
| 设备实时预处理 | 弱 | 强 | Electron 更适合靠近硬件 |

关键判断：

> Web 版天然适合“算法在服务器”；Electron 版天然适合“靠近设备、本地文件和本地算法”。

但从商业保护角度看：

> 即使用 Electron，核心算法也不建议完整下发到客户电脑。Electron 解决的是桌面能力问题，不解决核心算法防破解问题。

### 10.9 对内部 SDK 的架构影响

建议 SDK 不要做成 Web 一套、Electron 一套完全分裂的体系，而是拆成共享核心和平台适配层。

推荐结构：

```text
sdk-core
  业务对象、数据结构、校验规则、通用计算、类型定义

sdk-ui
  表单、表格、图表、报告、回放视图、人体区域图

sdk-api-client
  服务器接口、算法接口、文件接口、认证接口

sdk-web-runtime
  Web 路由、权限、缓存、上传下载、PWA 能力

sdk-electron-runtime
  本地文件、设备访问、本地算法桥接、自动更新、桌面菜单

sdk-agent-docs
  组件说明、组合规则、示例、禁用场景、PRD 到组件映射
```

这样做的好处：

- 业务对象和 UI 组件可以最大程度复用。
- Web 和 Electron 的差异集中在 runtime 层。
- Agent 生成软件时先选业务和组件，再根据交付形态选择运行时。
- 后续如果要做移动端、Tauri 或原生客户端，也不会推翻核心 SDK。

### 10.10 选型建议

建议采用：

> Web first，Electron for device/offline。

也就是：

- 默认所有内部工具、管理后台、报告系统、数据浏览系统都先做 Web 版。
- 只有明确需要本地设备、本地文件、离线运行、本地算法、低延迟采集时，才做 Electron 版。
- Electron 版尽量复用 Web 组件和业务逻辑，只增加桌面能力壳层。
- 核心算法仍优先放服务器或客户私有化服务器，Electron 只做必要的本地桥接和轻量能力。

### 10.11 具体场景判断表

| 场景 | 推荐形态 | 原因 |
| --- | --- | --- |
| 内部项目管理 | Web | 多人协作、权限、数据集中 |
| 样本数据浏览 | Web | 筛选、搜索、审核、分享更方便 |
| 报告生成和审核 | Web | 统一模板、统一版本、便于留痕 |
| 客户在线 Demo | Web | 打开链接即可体验 |
| 客户内网系统 | Web 私有化 | 集中部署，便于多人使用 |
| 传感器采集软件 | Electron | 需要本地设备和低延迟链路 |
| 设备标定工具 | Electron | 依赖本地硬件和驱动 |
| 本地视频压力回放 | Electron 或 Web | 如果文件大且离线，选 Electron；如果数据在服务器，选 Web |
| 批量文件转换 | Electron | 本地大文件处理更合适 |
| 无网销售演示 | Electron 或 PWA | 简单演示可 PWA，复杂演示选 Electron |
| 本地算法调试 | Electron | 需要调用本地环境和模型 |
| 正式算法服务 | Web/Electron 调服务器 | 核心算法放服务器更稳 |

### 10.12 第一版落地建议

第一版不要同时开发完整 Web 版和完整 Electron 版。建议先做以下拆分：

1. **先做 Web SDK 基座**
   - 业务对象。
   - UI 组件。
   - 算法 API 客户端。
   - 报告模板。
   - Agent 使用说明。

2. **再做一个最小 Electron 壳**
   - 复用 Web UI。
   - 增加本地文件读取。
   - 增加本地数据回放。
   - 预留本地算法桥接。
   - 不把核心算法下发到本地。

3. **用压力数据回放工具验证**
   - Web 版：适合服务器已有数据的浏览、筛选、报告。
   - Electron 版：适合客户现场导入本地视频和压力文件，离线回放。

推荐第一版验证目标：

- 同一套 `sdk-core` 和 `sdk-ui` 能同时服务 Web 和 Electron。
- Web 版能快速做业务管理和报告。
- Electron 版只补本地能力，不重写业务逻辑。
- Agent 能根据 PRD 判断应该生成 Web 版还是 Electron 版。

### 10.13 针对“读取硬件、可视化、采集、算法分析”的重新判断

如果公司大部分软件链路都是：

```text
读取硬件 -> 实时可视化 -> 数据采集 -> 算法分析 -> 报告/回放/管理
```

那么 Web 和 Electron 的区别不能只看“性能”。更准确的判断是：

> 可视化性能差异通常不是第一决策因素；硬件访问、采集稳定性、本地文件、离线能力和算法部署位置才是第一决策因素。

原因是 Electron 的界面层本质上也是 Chromium。也就是说，如果同样用 React/Vue + Canvas/WebGL 做压力热力图、曲线和时间轴，Web 版和 Electron 版的前端渲染能力不会天然差一个等级。真正的差异发生在数据进入界面之前：

- 数据怎么从硬件稳定读进来。
- 数据能不能长时间连续采集。
- 断网时能不能继续采。
- 原始数据能不能安全落本地文件。
- 算法是在服务器、本地服务，还是客户端进程里跑。
- 出问题时能不能拿到日志、重放数据和定位链路。

### 10.14 分链路对比

| 链路环节 | Web 版判断 | Electron 版判断 | 建议 |
| --- | --- | --- | --- |
| 读取硬件 | 受浏览器 API、权限、安全上下文和兼容性限制 | 可通过 Node.js、原生模块、本地服务、驱动 SDK 接入 | 强硬件链路优先 Electron |
| 实时可视化 | Canvas/WebGL/Web Worker 足够支撑多数场景 | 同样使用 Chromium，可复用 Web 可视化组件 | 不要因为“可视化性能”单独选 Electron |
| 数据采集 | 长时间采集、后台运行、本地落盘能力较弱 | 更适合持续采集、断点续采、本地文件和日志 | 正式采集工具优先 Electron |
| 算法分析 | 更适合调用服务器算法 API | 可调用服务器，也可桥接本地算法 | 核心算法仍建议服务器优先 |
| 数据管理 | 多人协作、权限、查询、报告更方便 | 单机数据管理容易分散 | 管理和报告优先 Web |
| 客户演示 | 在线演示方便 | 无网、现场、本地文件演示更稳 | 简单 Demo 用 Web，现场离线用 Electron |

### 10.15 性能要拆成四类看

讨论 Web 和 Electron 性能时，建议不要只说“哪个更快”，而要拆成四类。

#### 1. 硬件读取性能

这是 Web 的主要短板。

浏览器现在有 Web Serial、WebUSB、Web Bluetooth 等能力，但这些 API 普遍受安全上下文、用户授权、浏览器兼容性和设备协议限制。MDN 对 Web Serial 和 WebUSB 都标注了 limited availability，并说明相关能力只在安全上下文中可用；Web Bluetooth 也被标注为 experimental，需要谨慎看浏览器兼容性。

如果硬件是公司自研传感器、采集盒、串口设备、USB 设备、蓝牙设备或需要厂商驱动 SDK，Electron 更稳。

参考：

- [MDN Web Serial API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Serial_API)
- [MDN WebUSB API](https://developer.mozilla.org/en-US/docs/Web/API/WebUSB_API)
- [MDN Web Bluetooth API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Bluetooth_API)

#### 2. 实时可视化性能

这不是 Electron 的天然优势。

Web 和 Electron 都可以使用：

- Canvas。
- WebGL。
- Web Worker。
- OffscreenCanvas。
- 数据降采样。
- 环形缓冲区。
- 分片渲染。

如果压力数据是 100Hz、1024 点位，真正要优化的是数据管线和渲染策略，而不是先争论 Web/Electron。

典型策略：

- 原始数据完整保存。
- UI 只渲染当前窗口和必要降采样数据。
- 曲线只画屏幕可见点位，不把全部历史点每帧重绘。
- 热力图用 Canvas/WebGL，而不是用大量 DOM 节点。
- 采集线程、算法线程、UI 渲染线程分开。

参考：

- [MDN Web Workers API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API)
- [MDN OffscreenCanvas](https://developer.mozilla.org/en-US/docs/Web/API/OffscreenCanvas)

#### 3. 采集稳定性性能

这是 Electron 的主要优势。

正式采集软件通常需要：

- 长时间运行。
- 稳定连接设备。
- 自动重连。
- 本地缓存。
- 本地日志。
- 断网续传。
- 异常时保留原始数据。
- 调用系统能力或厂商 SDK。

这些能力用 Electron 或原生客户端更合适。Web 可以做轻量采集或 Demo，但不适合作为强依赖硬件的正式采集主链路。

#### 4. 算法性能

算法性能主要取决于算法放在哪里，而不是 Web/Electron 本身。

| 算法位置 | 适合形态 | 判断 |
| --- | --- | --- |
| 公司服务器 | Web/Electron 都适合 | 最利于保护算法和统一升级 |
| 客户私有化服务器 | Web/Electron 都适合 | 适合数据不能出内网的客户 |
| 本地轻量算法 | Electron 更适合 | 适合预处理、质量检查、离线预览 |
| 本地重算法 | Electron 可做但成本高 | 环境、驱动、GPU、版本管理复杂 |
| 浏览器 WASM | Web/Electron 都可尝试 | 适合轻量推理，不适合复杂依赖 |

关键结论：

> Electron 让“本地算法”更可行，但不代表核心算法应该放本地。核心算法放本地仍然有破解、泄露、版本失控和现场排障问题。

### 10.16 推荐的软件链路架构

结合当前业务，更推荐采用“三段式”架构：

```text
采集端
  Electron 或本地采集服务
  负责读取硬件、实时预览、原始数据落盘、断网缓存、基础质量检查

平台端
  Web
  负责项目管理、样本管理、数据浏览、报告审核、客户演示、权限协作

算法端
  公司服务器或客户私有化服务器
  负责核心算法、模型推理、报告生成、版本记录、调用统计
```

这样拆分后：

- Electron 不承担所有业务系统，只承担靠近硬件和本地文件的部分。
- Web 不强行读取所有硬件，只承担协作、管理、报告和分析入口。
- 算法不被绑死在客户端，方便保护、升级、审计和商业化。
- SDK 的 UI、数据结构、算法接口仍然可以复用。

### 10.17 两种可落地方案

#### 方案 A：Electron 采集端 + Web 管理分析端

这是更稳的推荐方案。

适合：

- 公司硬件协议会持续变化。
- 要做正式采集。
- 要支持离线和客户现场。
- 数据文件较大。
- 算法需要服务器统一管理。

链路：

```text
硬件 -> Electron 采集端 -> 本地原始数据/缓存 -> 上传 Web 平台 -> 算法服务 -> 报告/回放
```

优点：

- 采集稳定。
- 设备接入灵活。
- Web 平台便于协作和管理。
- 核心算法不下发。

缺点：

- 要维护一个桌面端。
- 要设计上传、同步和版本兼容机制。

#### 方案 B：Web 平台 + 本地采集网关

这是偏轻量的方案。

本地运行一个小型采集服务，Web 页面通过 `localhost` 或局域网地址连接这个服务。硬件读取由本地采集服务完成，Web 负责界面。

链路：

```text
硬件 -> 本地采集服务 -> Web 页面实时连接 -> 上传平台 -> 算法服务
```

优点：

- 主界面仍是 Web。
- 不需要完整 Electron 客户端。
- 更新 Web 页面方便。

缺点：

- 仍然要安装本地服务。
- 浏览器连接本地服务会带来权限、安全和端口占用问题。
- 本地服务的安装、升级、日志和排障仍要做。

适合：

- 采集端功能很薄。
- 硬件协议较稳定。
- 团队希望主应用尽量 Web 化。

### 10.18 当前建议

基于“读取硬件、可视化、采集、算法分析”这个链路，建议调整结论为：

> 不是 Web 和 Electron 谁性能更强，而是谁更适合承担哪一段链路。

具体建议：

1. **正式采集端优先 Electron**
   - 负责硬件读取、实时预览、本地缓存、原始数据保存、断网续传。

2. **业务平台优先 Web**
   - 负责样本管理、数据浏览、报告审核、客户演示、账号权限。

3. **核心算法优先服务器**
   - Web 和 Electron 都通过统一算法 API 调用，不把核心模型完整下发。

4. **可视化组件保持同一套**
   - 压力热力图、曲线、回放时间轴、人体区域图放在 `sdk-ui`，Web 和 Electron 共用。

5. **用性能测试验证，而不是凭感觉选型**
   - 用真实数据测 100Hz、1024 点位、30 分钟连续采集。
   - 指标包括丢帧率、延迟、CPU、内存、磁盘写入、可视化帧率、断线恢复、上传恢复。

第一版最合理的验证项目仍然是压力数据回放/采集工具：

- Electron 验证硬件读取、采集稳定、本地落盘和离线回放。
- Web 验证数据浏览、报告、算法服务调用和多人协作。
- 两端共用 SDK 数据结构和可视化组件。

### 10.19 实时算法与报告类算法要分开设计

你提到有些算法需要实时调用，一秒大几十 Hz；另一些算法是报告类算法。这个区别非常关键，建议 SDK 从第一版就把算法分成两类，而不是统一理解成“调用一个算法接口”。

```text
实时流式算法
  面向采集过程
  输入是连续传感器流
  输出是实时状态、质量提示、风险提示、动作阶段、预览结果

报告类批处理算法
  面向采集完成后
  输入是一段完整样本、清洗数据、标签、人工确认信息
  输出是分析结论、报告内容、评分、建议、质量结论
```

#### 实时流式算法

实时算法的核心指标不是“算得多复杂”，而是稳定、低延迟、可降级。

典型场景：

- 采集过程中判断设备是否正常。
- 判断压力点是否掉线、漂移、饱和。
- 实时显示动作阶段或关键事件。
- 实时提示采集质量是否合格。
- 实时生成热力图、区域压力曲线和简单特征。
- 对采集人员提示“需要重采”“动作不完整”“同步异常”。

特点：

- 调用频率高，例如 30-80Hz，甚至接近原始采样率。
- 输入是流式数据，不是一整个文件。
- 允许结果是临时判断，不一定作为最终报告结论。
- 必须有超时、跳帧、降采样、缓冲和断线恢复。
- 网络调用如果不稳定，会直接影响采集体验。

因此，实时算法更适合放在：

| 部署位置 | 适用程度 | 说明 |
| --- | --- | --- |
| Electron 本地进程 | 强 | 最适合低延迟、硬件贴近、断网可用 |
| 本地算法服务 | 强 | Python/C++/ONNX 可作为本地服务被 Electron 调用 |
| 客户内网边缘服务器 | 中 | 适合多设备共享算法，但要控制局域网延迟 |
| 公司云服务器 | 弱到中 | 只适合非关键实时提示，不适合强依赖采集闭环 |
| Web 浏览器内 | 弱到中 | 可做轻量特征和可视化，不适合重算法和硬件强绑定 |

实时算法建议分三层：

```text
L0 采集保护层
  设备连接、丢包、坏点、时间戳、缓存、落盘
  必须本地执行

L1 实时预览层
  热力图、曲线、简单滤波、基础特征、质量提示
  优先本地执行

L2 实时智能层
  动作阶段识别、异常动作提示、实时风险判断
  可本地轻量模型，也可边缘服务
```

实时算法的设计原则：

1. **不要每一帧都远程请求服务器**
   - 一秒几十 Hz 的远程请求会带来网络抖动、排队、费用和服务稳定性问题。
   - 服务器更适合接收片段、窗口或采集完成后的数据。

2. **用滑动窗口调用，而不是逐点调用**
   - 例如每 200ms、500ms 或 1s 处理一个窗口。
   - 原始数据仍按高频保存，算法只消费窗口。

3. **实时结果只做采集辅助，不直接当最终报告**
   - 实时提示可以帮助重采和现场判断。
   - 最终结论应由报告类算法基于完整数据重新计算。

4. **必须支持降级**
   - 算法慢了，UI 不能卡死。
   - 模型不可用时，采集仍要继续。
   - 实时智能结果缺失时，原始数据必须完整保存。

5. **实时算法要和 UI 解耦**
   - UI 订阅结果，不直接驱动算法循环。
   - 算法输出事件、状态和特征，UI 只负责展示。

#### 报告类批处理算法

报告类算法更适合放在服务器或客户私有化服务器。

典型场景：

- 采集完成后生成正式评估报告。
- 计算完整样本的统计特征。
- 结合多段数据、多指标和业务规则给出结论。
- 生成客户可读的解释文本。
- 记录算法版本、输入摘要、报告结论和审核状态。

特点：

- 不要求几十 Hz 实时响应。
- 可以接受几秒到几十秒的计算时间。
- 更重视结果一致性、可追溯、可复算。
- 更适合统一版本管理和权限控制。
- 更适合保护核心算法和商业规则。

报告类算法建议放在：

| 部署位置 | 适用程度 | 说明 |
| --- | --- | --- |
| 公司服务器 | 强 | 统一版本、保护算法、便于审计和计费 |
| 客户私有化服务器 | 强 | 适合数据不能出内网的客户 |
| Electron 本地 | 弱到中 | 适合离线演示或轻量报告，不建议放核心算法 |
| Web 浏览器内 | 弱 | 只适合简单计算和预览，不适合正式报告 |

报告类算法的设计原则：

1. **以完整样本为输入**
   - 包括原始数据、清洗数据、采集元信息、标签、人工确认信息。

2. **输出必须带版本信息**
   - 算法版本。
   - 模型版本。
   - 规则版本。
   - 输入数据摘要。
   - 生成时间。

3. **报告结论可复算**
   - 后续算法升级后，要能知道旧报告由哪个版本生成。
   - 必要时支持用新算法重新生成报告。

4. **正式结论应支持人工审核**
   - 尤其是评估、医疗、风险提示、客户交付报告。

### 10.20 对 Web/Electron 选型的修正

加入实时算法后，选型建议应进一步细化：

| 软件能力 | 推荐形态 | 原因 |
| --- | --- | --- |
| 硬件读取 | Electron | 更稳定接设备、驱动、本地 SDK |
| 高频实时算法 | Electron 本地或本地算法服务 | 避免网络抖动影响采集 |
| 实时可视化 | Web/Electron 共用组件 | 渲染能力主要看 Canvas/WebGL 和数据管线 |
| 原始数据采集落盘 | Electron | 更适合本地文件、断网缓存、异常恢复 |
| 采集质量实时提示 | Electron | 要靠近采集链路 |
| 正式报告算法 | 服务器或客户私有化服务器 | 统一版本、审计、保护核心算法 |
| 报告查看和审核 | Web | 多人协作、权限、分享、留痕 |
| 数据管理和客户项目 | Web | 集中管理更合适 |
| 离线演示 | Electron | 可带轻量算法和样例数据 |

因此，最推荐的软件链路改成：

```text
硬件
  -> Electron 采集端
    -> 本地实时算法/质量检查/可视化
    -> 原始数据落盘
    -> 上传 Web 平台
      -> 服务器报告算法
      -> 报告审核/管理/分享
```

其中实时算法和报告算法的关系是：

```text
实时算法：帮你把数据采好
报告算法：帮你把数据解释清楚
```

这两个算法可以共用部分基础特征计算，但不建议完全共用同一个运行方式。

### 10.21 SDK 算法接口建议

建议 `SDK算法接口规范.md` 里不要只定义一个 `runAlgorithm()`，而是定义两套接口。

#### 实时算法接口

```text
RealtimeAlgorithm.start(config)
RealtimeAlgorithm.pushFrame(frame)
RealtimeAlgorithm.pushWindow(window)
RealtimeAlgorithm.onResult(callback)
RealtimeAlgorithm.onWarning(callback)
RealtimeAlgorithm.stop()
```

实时算法输出建议是事件流：

```text
quality_warning
device_warning
motion_phase_changed
feature_updated
preview_updated
latency_warning
```

实时算法必须有：

- 最大处理延迟。
- 输入频率上限。
- 可接受丢帧策略。
- 是否影响采集继续。
- 本地日志路径。
- 算法版本。

#### 报告算法接口

```text
ReportAlgorithm.createJob(sampleId, options)
ReportAlgorithm.getJobStatus(jobId)
ReportAlgorithm.getResult(jobId)
ReportAlgorithm.regenerate(reportId, version)
```

报告算法输出建议是结构化结果：

```text
summary
metrics
charts
quality_result
risk_level
recommendations
algorithm_version
model_version
input_hash
review_required
```

报告算法必须有：

- 输入数据版本。
- 算法版本。
- 生成时间。
- 可复算标记。
- 审核状态。
- 错误原因和重试机制。

### 10.22 性能验证指标

针对实时算法，一定要做真实压力测试。建议第一版就定义这些指标：

| 指标 | 建议目标 |
| --- | --- |
| 输入频率 | 按真实硬件，例如 100Hz 原始采样 |
| 实时算法频率 | 按需求，例如 30-80Hz 或窗口级处理 |
| 端到端延迟 | 采集到 UI 提示尽量 < 100-200ms，具体按业务体验验证 |
| 丢帧策略 | UI 可降采样，原始数据不能丢 |
| 连续采集时长 | 至少 30 分钟稳定运行 |
| CPU/内存 | 采集期间不能持续爬升 |
| 本地落盘 | 异常退出后能保留已采数据 |
| 断网行为 | 采集不中断，恢复后可上传 |
| 算法异常 | 实时算法失败不能导致原始采集中断 |

针对报告算法，指标不同：

| 指标 | 建议目标 |
| --- | --- |
| 生成耗时 | 按报告复杂度定义，可接受数秒到数十秒 |
| 结果一致性 | 同一输入和算法版本应生成同一结果 |
| 可追溯 | 能追踪算法版本、模型版本、输入摘要 |
| 可复算 | 支持重新生成报告 |
| 权限和审核 | 正式报告支持人工审核和留痕 |
| 失败恢复 | 支持任务重试、错误提示、后台队列 |

## 12. 第一版里程碑

### 阶段 1：沉淀文档和边界

目标：

- 完成 SDK 调研文档。
- 完成 PRD 模板。
- 完成第一版组件清单。
- 选择 1-2 个真实业务场景作为样例。

建议样例：

- 压力数据同步回放工具。
- 肌少症/身体功能评估报告工具。

### 阶段 2：封装最小可用组件

目标：

- 封装基础数据结构。
- 封装 5-10 个高频 UI 组件。
- 封装 2-3 个算法调用接口。
- 提供可运行示例。

验收标准：

- Agent 能根据文档生成一个简单业务工具。
- 生成结果能复用 SDK 组件，而不是从零写页面。
- 业务负责人能看懂输入输出和边界。

### 阶段 3：接入算法服务和版本管理

目标：

- 建立统一算法 API。
- 支持本地/服务器两种调用实现。
- 算法结果带版本号和日志。
- 支持基础授权和调用统计。

验收标准：

- 同一业务软件可以切换本地算法和服务器算法。
- 算法升级不需要重写业务页面。
- 可以追踪某个报告由哪个算法版本生成。

## 13. 关键风险

| 风险 | 表现 | 建议 |
| --- | --- | --- |
| SDK 做得太大 | 长期开发但没人用 | 从 1-2 个真实场景开始 |
| 组件太抽象 | Agent 不知道怎么组合 | 每个组件配具体示例 |
| 文档只给人看 | Agent 读不出约束 | 写清参数、输入、输出、禁用场景 |
| 算法本地化过深 | 环境难装、容易泄露 | 核心算法服务器优先 |
| 服务器依赖过重 | 断网不能用 | 保留本地轻量版和私有化方案 |
| PRD 模板太复杂 | 非业务人员不愿填 | 模板先覆盖关键问题，不追求完整 |
| 业务口径无人确认 | 软件能跑但结果不可信 | 每个业务包指定口径负责人 |

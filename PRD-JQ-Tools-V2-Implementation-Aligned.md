# JQ Tools V2.0 压力感知软件 PRD（当前实现对齐版）

> 文档日期：2026-05-08  
> 文档类型：研发交付 PRD  
> 基线来源：已实现功能、原 V2.0 PRD、`shroom/FUNCTION_IMPLEMENTATION.md`、`opt.md`  
> 适用对象：产品、研发、测试、算法、硬件、交付、售后

## 1. Summary

JQ Tools V2.0 是一套用于压力感知硬件的桌面软件，面向汽车座椅、靠背、床垫、坐垫、手部、脚部等压力传感器场景。软件已具备设备连接、实时可视化、框选、量尺、数据统计、采集、历史回放、CSV 导入导出、设备管理和 Windows/macOS 打包能力。

本 PRD 以当前已经实现的功能为基线，重新定义 V2.0 的交付范围、验收标准和待补齐项。目标不是重新规划一个理想版本，而是让研发和 QA 可以基于现有实现继续补齐数据口径、异常逻辑和交付验收。

## 3. Background

### 3.1 产品背景

压力感知系统由 JQ Tools 软件和适配传感器构成，用于从设备端采集压力数据，并在软件端实时展示、分析、采集、回放和导出压力数据。

当前业务重点是汽车座椅分析系统，尤其是座椅和靠背的压力实时反馈、压力图像展示、数据采集和科研分析。产品需要对标 XSENSOR、Tekscan 等国外品牌，在成本、适配性、本地化、可视化和数据采集能力上形成替代价值。

### 3.2 为什么需要重写 PRD

原 PRD 偏规划，`FUNCTION_IMPLEMENTATION.md` 偏代码实现，二者存在一些口径差异：

| 差异点 | 当前问题 |
|---|---|
| 设备类型识别 | PRD 写固件/设备信息自动返回，当前实现更依赖 MAC 本地映射。 |
| 缩放范围 | PRD 写 10%-1000%，当前实现文档写 10%-300%。 |
| 原始数据保存 | PRD 要求保留全量原始数据，当前实现存在翻转后落库逻辑，需要确认。 |
| 置零数据流 | 当前实现有置零功能，但对统计、采集、导出的影响需要明确。 |
| CSV 字段 | 当前导出字段和 PRD 标准字段不完全一致。 |
| 异常恢复 | 已有 SQLite/WAL/缓存能力，但恢复流程需要产品化定义。 |

因此，本 PRD 重新按照“当前实现 + 需要补齐”的方式组织，减少研发和 QA 的理解成本。

## 4. Objective

### 4.1 产品目标

V2.0 的目标是完成一套可交付的压力感知桌面软件，使用户可以完成从设备连接到数据采集、分析、回放和导出的完整闭环。

### 4.2 业务目标

1. 支撑汽车座椅/靠背压力感知系统的展示、测试和交付。
2. 支撑门店、展厅、研发和科研客户的实时压力图像展示。
3. 提供可追溯、可导出、可回放的数据能力。
4. 为后续数据对比、报告生成、标定分析和高级算法打基础。

### 4.3 V2.0 成功标准

| 指标 | 成功标准 |
|---|---|
| 设备连接 | 合法设备可一键连接，非法设备不能采集。 |
| 实时展示 | 2D/3D 视图可实时反映压力变化。 |
| 数据采集 | 可开始、停止、命名、备注并保存历史记录。 |
| 数据回放 | 历史数据可播放、暂停、拖动进度、倍速回放。 |
| 数据导出 | CSV 可导出，并可被软件重新导入校验。 |
| 数据可信 | 原始数据、显示数据、统计数据、导出数据口径明确。 |
| 稳定性 | 采集中异常退出后，可恢复或保留已采集数据。 |
| 兼容性 | Windows/macOS 可打包运行，串口缓存写入 userData。 |

## 7. Solution

### 7.1 版本范围定义

本 PRD 将功能分成三类：

| 状态 | 含义 |
|---|---|
| 已实现 | 当前代码或实现文档已经明确具备。 |
| V2.0 待补齐 | 当前已部分实现，但文档、逻辑、字段或验收口径需要补齐。 |
| V2.1+ | 不阻塞当前 V2.0，但建议作为后续版本规划。 |

### 7.2 主流程

```text
打开软件
-> 检查 MAC/设备配置
-> 一键连接设备
-> 读取 MAC/设备类型/采样率/串口信息
-> 进入实时展示
-> 使用 2D/3D 视图和分析工具
-> 开始采集
-> 停止采集并保存历史
-> 回放历史数据
-> 导出或导入 CSV
-> 后续进入数据对比
```

| 待补齐项 | 优先级 | 说明 |
|---|---|---|
| 设备类型识别口径 | P0 | 明确当前是 MAC 本地映射，不是完全由固件自动返回。 |

### 7.3.5 验收标准

1. 无设备配置时，启动后进入 MAC 配置页。
2. 合法设备可一键连接成功。
3. 非法设备、类型不匹配设备不能开始采集。
4. 连接成功后能看到 MAC、设备类型、串口、波特率、采样率。
5. 设备断开后界面有明确状态变化和提示。

### 7.6.5 验收标准

## 7.9 全局数据规则

### 7.9.1 数据分层

为避免后续数据口径混乱，V2.0 必须明确以下数据层：

| 数据层 | 定义 | 是否必须保留 |
|---|---|---|
| `rawData` | 设备原始方向、原始压力矩阵 | 必须保留 |
| `zeroedData` | 扣除置零基线后的矩阵 | 根据置零状态生成 |
| `displayData` | 用于界面展示的矩阵，可受翻转、颜色、润滑影响 | 必须可重建 |
| `calculatedData` | 用于统计指标的矩阵 | 口径必须明确 |
| `storedData` | 采集入库数据 | 必须可追溯 |
| `exportedData` | CSV 导出数据 | 必须标明口径 |

### 7.9.2 原始数据原则

1. 框选不得导致全量原始矩阵丢失。
2. 可视化调节不得污染原始数据。
3. 翻转如参与落库，必须保存 `dataDirection`，并保证可还原。
4. 置零如影响统计或导出，必须记录置零状态和置零时间。
5. CSV 导出必须让后续导入、回放和对比有足够字段。

## 7.11 Assumptions

| 假设 | 风险 | 验证方式 |
|---|---|---|
| MAC 映射可以满足当前设备类型识别 | 固件未返回类型时仍可用，但配置成本高 | 用真实设备测试首次配置和批量录入 |
| 当前缩放 300% 已满足使用 | 如果客户需要查看单点，可能不够 | 用展厅和研发场景确认 |
| CSV 当前字段能支撑科研分析 | 字段不标准会影响后续导入/对比 | 用导出 CSV 做二次导入测试 |
| 置零逻辑已满足业务 | 如果只影响显示，会影响数据可信 | QA 按置零前后矩阵验证 |
| 异常恢复能力足够 | 采集中断可能造成历史记录不完整 | 做强退和断电模拟测试 |
